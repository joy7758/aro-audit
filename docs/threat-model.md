# Threat Model

This document scopes the current threat model for ARO Audit as an Audit
Evidence Layer candidate. It focuses on what the repository can already
evidence, what it can partially support, and what remains future work.

| Threat | Failure mode | Control expectation | Evidence expected | Current repo status | Gap / next step |
| --- | --- | --- | --- | --- | --- |
| memory poisoning | Upstream memory or context injects false state that later appears in a receipt or journal as if it were trusted runtime truth. | High-risk context should be narrowed before handoff, and the record should preserve what was declared rather than silently rewriting it. | Compact receipt fields such as actor, model, tools, input summary, output summary, plus any review notes about source boundaries. | Partial. Example receipts preserve emitted summaries and actor/tool declarations, but this repo does not claim memory sanitization or source provenance enforcement. | Add candidate fixtures that show poisoned-context review cases and clearer provenance annotations for bounded verification surface review. |
| tool misuse | An agent calls the wrong tool, or uses an allowed tool outside the intended task boundary. | Tool use should be visible in the review artifact, and downstream reviewers should be able to compare declared tools with observed action records. | Tool lists in receipts, action records in journals, and reviewer-visible policy signals when available. | Partial. The repo can record tools in receipt examples and journals in quickstart flows, but it is not the default tool-enforcement layer. | Add candidate fixtures linking declared tool scope to exportable evidence and optional governance adapters. |
| privilege compromise | A run executes with more authority than was intended, or reuses a privileged context without a clear review trail. | Privilege assumptions should be explicit in the evidence handoff, and governance-linked inputs should be reviewable when present. | Actor identity, task boundary, policy signals, governance-linked metadata, and reviewable run records. | Partial. The repo can package actor and policy-facing fields in review artifacts, but it does not by itself enforce privilege boundaries. | Add candidate evidence fixtures that show privilege scope, escalation attempts, and handoff from governance controls into the audit layer. |
| cascading hallucinations | A misleading output is repeated across steps and later treated as authoritative because it was logged. | Evidence should preserve what happened without implying that the repo verified factual correctness. Reviewers need a bounded verification surface, not a truth oracle. | Input/output summaries, outcome fields, trace or receipt summaries, and reviewer annotations when available. | Partial. The repo records output-facing summaries and review artifacts, but it does not claim semantic truth verification. | Add candidate hardening fixtures for factual drift and reviewer guidance on what evidence can and cannot prove. |
| intent breaking | The declared task or predicate is changed so the evidence no longer matches the original intent boundary. | Fact-bearing boundary fields should fail verification when mutated, and reviewer-visible intent fields should remain stable across the record. | Conformance vector failure output, preserved task or predicate fields, and verifier-visible mismatch signals. | Present for fixed boundary tamper vectors. `spec/test_vectors/boundary_attack_predicate.jsonl` is expected to fail, showing current support for predicate-boundary mutation rejection. | Add compact intent-drift fixtures that connect task summaries, predicates, and receipt review signals across more scenarios. |
| replay / stale receipt reuse | An old receipt, journal, or checkpoint is replayed as if it were fresh, or reused outside its intended review window. | Review artifacts should preserve timestamps, sequence continuity, and chain linkage. Freshness policy should be explicit where required. | Timestamp fields, sequence continuity, checkpoint linkage, verifier output, and any future freshness annotations. | Partial. Quickstart and chain verification cover tamper detection and linkage checks, but the repo does not yet define a general freshness policy for stale receipt reuse. | Add timestamp replay vectors, freshness guidance, and candidate fixtures for stale artifact handoff. |
